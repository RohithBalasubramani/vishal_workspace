Part 1 — Non-Technical Overview

1.1 What the System Does

This system reads electrical product catalog PDFs from manufacturers like Lauritz Knudsen, Schneider Electric, ABB, and Siemens. It extracts every product's specifications — model numbers, current ratings, breaking capacities, pole configurations, curve types, prices — and organizes them into a searchable database.

Think of it as a specialist who can read a 4-page catalog with hundreds of rows, understand the tables and product images, and enter every product into a perfectly organized database in under a minute — except it does this automatically, consistently, and can be asked questions about the data afterward.

Key Capabilities:

  • Reads catalog pages using OCR (via the upstream OCR Pipeline v2)
  • Extracts product specifications from tables using an AI language model (Qwen 3.5-27B)
  • Stores products in a structured PostgreSQL database with flexible key-value specs
  • Extracts product images from PDFs and links them to the correct products using text-proximity matching
  • Provides a 5-tab web interface for uploading, browsing, editing, chatting, and exporting
  • Detects product categories (MCB, DC MCB, HR MCB, Solar Combo, MCCB, RCCB) intelligently
  • Supports benchmarking extraction quality between Qwen and Claude models

1.2 How It Works (Plain English)

Step 1 — Upload Catalog: A catalog PDF is uploaded through the web interface or provided via the command line. The system accepts catalogs from any electrical product manufacturer.

Step 2 — Read Pages (OCR): Each PDF page is scanned by the upstream OCR Pipeline (~/mcb_test/ocr_complete) which uses GPU-powered AI (DeepSeek-OCR-2) to convert images of text and tables into machine-readable data.

Step 3 — Extract Products: The OCR'd tables are sent to an AI language model (Qwen 3.5-27B running locally via vLLM) which understands the table structure and extracts every product row with its complete specifications.

Step 4 — Save to Database: Products are inserted into PostgreSQL with duplicate detection (by model number). Each product gets 11-14 key-value specification pairs.

Step 5 — Extract & Link Images: Product images are extracted from the PDF using PyMuPDF. Each image is linked to the correct product by finding where the product's model number appears on the page and assigning the nearest image.

Step 6 — Browse & Query: Users can search products by specifications, browse by category, edit entries, ask the AI chatbot questions, or export data as CSV/Excel.

1.4 Web Interface

The web interface (Gradio) runs at http://localhost:7862 and has five tabs:

  • Upload & Extract — Upload a catalog PDF; the pipeline runs OCR, LLM extraction, DB insert, and image linking with real-time streaming progress updates.
  • Browse Products — Filter by category, search by product name or model number, and view detailed specs for any product.
  • Review & Edit — Edit product name/model/category, delete products, or add new specification key-value pairs.
  • Ask AI — Chat with an AI assistant that searches the product database for relevant context and answers questions about products, prices, and specifications.
  • Export — Download the full catalog as CSV or Excel (.xlsx) with three sheets: Products, Specs (Pivot), and Specs (Raw).

2.3 Extraction Rules

The LLM extraction uses a single method: Qwen 3.5-27B via vLLM with a structured prompt.

ParameterValue
ModelQwen/Qwen3.5-27B-FP8
Temperature0.05 (near-deterministic)
Max Tokens4096–8192
Timeout120 seconds
Thinking ModeDisabled (enable_thinking: false)
Max Rows per Call50 (tables split if larger)

Extraction Prompt Rules:

  • Extract EVERY product row — never sample or summarize
  • Required output fields: product_name, product_model, category, specs (dict)
  • Categories: MCB, MCCB, RCCB, RCBO, Contactor, Relay, Switch, Fuse, SPD, Isolator, Starter, Other
  • Capture all specs: rating, poles, voltage, price, curve type, breaking capacity, dimensions, standards
  • Handle multiple currencies: INR, USD, EUR, or unlabeled
  • Output must be valid JSON array

Brand Detection: The system auto-detects brand from the filename. Supported keywords: Schneider, Siemens, ABB, Legrand, Hager, Mitsubishi, Lauritz, LK, Havells, Polycab, L&T.

Duplicate Handling: Products are deduplicated by the product_model UNIQUE constraint. Existing models are skipped, not overwritten.

2.4 Image Extraction & Linking

The image pipeline extracts embedded product photos from PDFs and assigns each one to the correct product using text-proximity matching.

Image Filtering Rules

FilterThresholdPurpose
Minimum size100 × 100 pxSkip icons, bullets, logos
Minimum area15,000 px²Skip small decorative graphics
Maximum aspect ratio5:1Skip banners, strips, headers
Full-page filter>90% of page sizeSkip background images
Product image filter<400,000 px²Skip large promotional graphics

Text-Proximity Matching Algorithm

Instead of assigning images by page order or heuristic rules, the system uses the actual position of product model numbers in the PDF text to find the nearest image:

  1. Build text index: Scan each PDF page using page.get_text("dict") to extract every text span with its Y-coordinate.
  2. Locate product: For each product in the database, search the text index for its model number (e.g., "BB10010C"). This gives the page number and Y-position where the product appears in the catalog.
  3. Filter images: On the same page, filter to only product-quality images (not backgrounds, banners, or decorative elements).
  4. Find nearest: Calculate the vertical distance from the product's text position to each image's center point. Assign the closest image.
  5. Fallbacks: If the model number isn't found, try the catalog_number field. If still no match, assign the first available product image.

Result: Products in different table sections (SP, DP, TP, FP) on the same page are correctly assigned to their respective product images based on physical proximity in the catalog layout.

2.5 AI Chat (RAG)

The Ask AI tab provides a Retrieval-Augmented Generation chatbot:

ParameterValue
ModelQwen/Qwen3.5-27B-FP8
Temperature0.1
Max Tokens1024
Context Limit10 products per query
StreamingYes (token-by-token)

Retrieval Process:

  1. Extract keywords from the user's question (words > 2 characters)
  2. Search products by: product_name ILIKE, product_model ILIKE, or spec_value ILIKE
  3. Build context string: product name, model, category, and up to 10 specs per product
  4. Send to LLM with system prompt: "Answer using ONLY the provided product data. Be precise with specs and prices."
  5. Stream response back to the chat interface

3.6 CLI Reference & Deployment

Command-Line Interface:

python main.py                                  # Launch Gradio UI on port 7862
python main.py --port 8000                      # Launch UI on custom port
python main.py --init-db                        # Initialize database schema
python main.py --seed                           # Seed DB with ~256 LK MCB products
python main.py --extract /path/to/catalog.pdf   # Extract products from PDF
python main.py --extract catalog.pdf --brand ABB # Extract with brand hint

Deployment:

# 1. Ensure PostgreSQL is running
sudo systemctl start postgresql

# 2. Ensure vLLM is serving Qwen 3.5-27B
# (on localhost:8200)

# 3. Initialize and seed
python main.py --init-db
python main.py --seed

# 4. Launch the application
python main.py --port 7862

# 5. (Optional) Expose via Cloudflare Tunnel
cloudflared tunnel --url http://localhost:7862
